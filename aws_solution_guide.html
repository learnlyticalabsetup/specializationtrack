Assignment 1: AWS VPC and Networking Setup
üåê VPC & Networking ‚è±Ô∏è 90 minutes üìä Foundational
‚è±Ô∏è Estimated Time: 90 minutes
üéØ Learning Objective: Create a production-ready VPC with public and private subnets

üìã Prerequisites

  • AWS Account with appropriate permissions
  • Basic understanding of IP addressing and CIDR notation
  • Familiarity with networking concepts (subnets, routing, gateways)
  • AWS Management Console access
Step 1: Creating the VPC
Set up the foundational network infrastructure
1
Access VPC Dashboard

Log into the AWS Management Console:

  1. Navigate to https://console.aws.amazon.com
  2. Sign in with your AWS credentials
  3. In the services menu, search for "VPC"
  4. Click on "VPC" under Networking & Content Delivery
üí° Quick Access: You can also use the search bar at the top and type "VPC" for faster navigation.
2
Create the VPC

In the VPC Dashboard:

  1. Click "Create VPC" button
  2. Select "VPC and more" for a comprehensive setup
  3. Configure the VPC settings:
Resources to create: VPC and more Name tag auto-generation: Auto-generate IPv4 CIDR block: 10.0.0.0/16 Number of Availability Zones: 2 Number of public subnets: 2 Number of private subnets: 2 NAT gateways: 1 per AZ VPC endpoints: None DNS hostnames: Enable DNS resolution: Enable
‚ö†Ô∏è Cost Notice: NAT Gateways incur charges (~$45/month each). For learning, you can start with 1 NAT Gateway and add more later.
3
Review and Create

Review your configuration:

  • VPC CIDR: 10.0.0.0/16 (65,536 IP addresses)
  • Public Subnets: 10.0.1.0/24 and 10.0.2.0/24
  • Private Subnets: 10.0.3.0/24 and 10.0.4.0/24
  • Availability Zones: Two different AZs for high availability

Click "Create VPC" and wait for creation to complete (usually 2-3 minutes).

‚úÖ Success: You should see "Successfully created VPC" message with all resources listed.
Step 2: Configuring Security Groups
Create security groups to control traffic flow
1
Create Web Server Security Group

In the VPC Dashboard:

  1. Click "Security Groups" in the left navigation
  2. Click "Create security group"
  3. Configure the web server security group:
Security group name: WebServer-SG Description: Security group for web servers VPC: Select your newly created VPC Inbound Rules: Rule 1: Type: HTTP Protocol: TCP Port Range: 80 Source: 0.0.0.0/0 (Anywhere) Rule 2: Type: HTTPS Protocol: TCP Port Range: 443 Source: 0.0.0.0/0 (Anywhere) Rule 3: Type: SSH Protocol: TCP Port Range: 22 Source: Your IP address (will auto-populate)

Click "Create security group".

2
Create Database Security Group

Create a more restrictive security group for databases:

Security group name: Database-SG Description: Security group for database servers VPC: Select your newly created VPC Inbound Rules: Rule 1: Type: MySQL/Aurora Protocol: TCP Port Range: 3306 Source: Custom (select WebServer-SG from dropdown) Rule 2: Type: PostgreSQL Protocol: TCP Port Range: 5432 Source: Custom (select WebServer-SG from dropdown)
üí° Security Best Practice: Database security groups should only allow access from application servers, never directly from the internet.
3
Create Application Load Balancer Security Group

Create a security group for the load balancer:

Security group name: ALB-SG Description: Security group for Application Load Balancer VPC: Select your newly created VPC Inbound Rules: Rule 1: Type: HTTP Protocol: TCP Port Range: 80 Source: 0.0.0.0/0 (Anywhere) Rule 2: Type: HTTPS Protocol: TCP Port Range: 443 Source: 0.0.0.0/0 (Anywhere)
Step 3: Setting Up Network ACLs
Configure additional subnet-level security
1
Create Custom Network ACL

Add an extra layer of security:

  1. Click "Network ACLs" in the VPC dashboard
  2. Click "Create network ACL"
  3. Configure:
Name: WebTier-NACL VPC: Select your VPC

Click "Create network ACL".

2
Configure NACL Rules

Add inbound rules to the NACL:

Inbound Rules: Rule 100: Type: HTTP (80) Source: 0.0.0.0/0 Allow/Deny: Allow Rule 110: Type: HTTPS (443) Source: 0.0.0.0/0 Allow/Deny: Allow Rule 120: Type: SSH (22) Source: Your IP/32 Allow/Deny: Allow Rule 130: Type: Custom TCP Port Range: 32768-61000 Source: 0.0.0.0/0 Allow/Deny: Allow
üí° Ephemeral Ports: Rule 130 allows response traffic on ephemeral ports that applications use for return traffic.
3
Associate NACL with Subnets

Link the NACL to public subnets:

  1. Select your newly created NACL
  2. Go to "Subnet associations" tab
  3. Click "Edit subnet associations"
  4. Select both public subnets
  5. Click "Save changes"
Step 4: Testing Network Connectivity
Verify that network routing and connectivity work correctly
1
Launch Test Instance in Public Subnet

Create a test EC2 instance:

  1. Go to EC2 Dashboard
  2. Click "Launch Instance"
  3. Configure:
Name: PublicSubnet-Test AMI: Amazon Linux 2023 AMI (Free Tier) Instance Type: t2.micro Key Pair: Create new or use existing Network Settings: VPC: Your VPC Subnet: Public subnet (10.0.1.0/24) Auto-assign public IP: Enable Security Group: WebServer-SG
2
Test Internet Connectivity

Verify the instance can reach the internet:

  1. Wait for instance to be in "Running" state
  2. Select the instance and click "Connect"
  3. Use "Session Manager" or SSH to connect
  4. Run connectivity tests:
ping -c 4 8.8.8.8
curl -I https://aws.amazon.com
‚úÖ Expected Result: Both commands should complete successfully, indicating internet connectivity.
3
Launch Test Instance in Private Subnet

Test the private subnet configuration:

Name: PrivateSubnet-Test AMI: Amazon Linux 2023 AMI Instance Type: t2.micro Network Settings: VPC: Your VPC Subnet: Private subnet (10.0.3.0/24) Auto-assign public IP: Disable Security Group: Database-SG

Connect via Session Manager and test that it can reach the internet through NAT Gateway:

ping -c 4 8.8.8.8
üí° Troubleshooting: If the private instance can't reach the internet, check that the NAT Gateway is properly configured and route tables are correct.

üîç Validation Checklist

  1. VPC Created: Verify VPC exists with correct CIDR (10.0.0.0/16)
  2. Subnets: 2 public and 2 private subnets in different AZs
  3. Internet Gateway: Attached to VPC and routes configured
  4. NAT Gateway: Created in public subnet with proper routing
  5. Security Groups: All 3 security groups created with correct rules
  6. Network ACL: Custom NACL associated with public subnets
  7. Connectivity: Public instance has internet access, private instance has outbound access via NAT

üõ†Ô∏è Common Issues & Solutions

  • Instance can't connect to internet: Check route table has 0.0.0.0/0 pointing to Internet Gateway
  • Private instance no internet: Verify NAT Gateway exists and private route table points to it
  • SSH connection fails: Ensure security group allows SSH (port 22) from your IP
  • High NAT Gateway costs: Consider using single NAT Gateway for learning environments

Step 1: Creating Launch Template
Define the configuration for EC2 instances
1
Access EC2 Launch Templates

In the AWS Console:

  1. Navigate to EC2 Dashboard
  2. In the left menu, click "Launch Templates"
  3. Click "Create launch template"
2
Configure Launch Template

Set up the template with these specifications:

Launch template name: WebServer-LaunchTemplate Template version description: Initial version for web servers Application and OS Images: AMI: Amazon Linux 2023 AMI (ami-xxxxxxxxx) Instance type: t3.micro Key pair: Create new or select existing Network settings: Subnet: Don't include in launch template Security groups: WebServer-SG (from Assignment 1) Storage (volumes): Volume 1 (AMI Root): 8 GiB, gp3, Encrypted Resource tags: Key: Name, Value: WebServer-AutoScaling Key: Environment, Value: Production
3
Add User Data Script

In the "Advanced details" section, add this user data script:

#!/bin/bash yum update -y yum install -y httpd systemctl start httpd systemctl enable httpd # Create a simple web page with instance information echo "Web Server" > /var/www/html/index.html echo "

Hello from $(hostname -f)

" >> /var/www/html/index.html echo "

Instance ID: $(curl -s http://169.254.169.254/latest/meta-data/instance-id)

" >> /var/www/html/index.html echo "

Availability Zone: $(curl -s http://169.254.169.254/latest/meta-data/placement/availability-zone)

" >> /var/www/html/index.html echo "

Private IP: $(curl -s http://169.254.169.254/latest/meta-data/local-ipv4)

" >> /var/www/html/index.html echo "

Server Time: $(date)

" >> /var/www/html/index.html echo "" >> /var/www/html/index.html # Install CloudWatch agent yum install -y amazon-cloudwatch-agent

Click "Create launch template".

Step 2: Creating Auto Scaling Group
Set up automatic scaling based on demand
1
Create Auto Scaling Group

In the EC2 Dashboard:

  1. Click "Auto Scaling Groups" in the left menu
  2. Click "Create Auto Scaling group"
  3. Configure the auto scaling group:
Auto Scaling group name: WebServer-ASG Launch template: WebServer-LaunchTemplate Version: Latest

Click "Next".

2
Configure Network Settings

Set up network and availability zones:

VPC: Your VPC (from Assignment 1) Availability Zones and subnets: ‚úÖ Select both public subnets: - Public subnet 1 (10.0.1.0/24) - Public subnet 2 (10.0.2.0/24)
üí° High Availability: Using multiple AZs ensures your application remains available even if one AZ experiences issues.
3
Configure Group Size and Scaling

Set up the scaling parameters:

Group size: Desired capacity: 2 Minimum capacity: 1 Maximum capacity: 4 Scaling policies: Target tracking scaling policy: ‚úÖ Enabled Metric type: Average CPU utilization Target value: 70 Instance scale-in protection: Disabled

This configuration will maintain 2 instances normally, but can scale up to 4 if CPU usage exceeds 70%.

Step 3: Setting Up Application Load Balancer
Distribute traffic across multiple instances
1
Create Application Load Balancer

In the EC2 Dashboard:

  1. Click "Load Balancers" in the left menu
  2. Click "Create Load Balancer"
  3. Choose "Application Load Balancer"
  4. Configure:
Load balancer name: WebServer-ALB Scheme: Internet-facing IP address type: IPv4 Network mapping: VPC: Your VPC Mappings: Select both Availability Zones - Public subnet 1 - Public subnet 2 Security groups: Remove default security group Add: ALB-SG (from Assignment 1)
2
Configure Target Group

Create a target group for the load balancer:

Target type: Instances Target group name: WebServer-TG Protocol: HTTP Port: 80 VPC: Your VPC Health checks: Health check protocol: HTTP Health check path: / Healthy threshold: 2 Unhealthy threshold: 5 Timeout: 5 seconds Interval: 10 seconds Success codes: 200

Don't register targets yet - the Auto Scaling Group will handle this.

3
Complete Load Balancer Creation

Finish the load balancer setup:

  1. Review all settings
  2. Click "Create load balancer"
  3. Wait for the load balancer to become "Active" (2-3 minutes)
‚úÖ Verification: Load balancer state should show "Active" and all target groups should be healthy.
Step 4: Integrating Auto Scaling with Load Balancer
Connect the auto scaling group to the load balancer
1
Update Auto Scaling Group

Modify the ASG to use the load balancer:

  1. Go to Auto Scaling Groups
  2. Select "WebServer-ASG"
  3. Go to "Details" tab
  4. Click "Edit"
Load balancing: ‚úÖ Application Load Balancer target groups Choose from your load balancer target groups: WebServer-TG Health checks: Health check type: ELB Health check grace period: 300 seconds

Click "Update".

2
Verify Instance Registration

Check that instances are registering with the target group:

  1. Go to Load Balancers > WebServer-ALB
  2. Click on "Target groups" tab
  3. Click on "WebServer-TG"
  4. Check the "Targets" tab

You should see your ASG instances listed with "healthy" status after a few minutes.

üí° Health Check Time: Initial health checks may take 2-3 minutes to show "healthy" status.
3
Test Load Balancer

Verify the load balancer is distributing traffic:

  1. Get the load balancer DNS name from the ALB details
  2. Open the DNS name in a web browser
  3. Refresh the page multiple times
  4. You should see different instance IDs and AZs
curl http://your-alb-dns-name.region.elb.amazonaws.com
‚úÖ Expected Result: Each refresh should show different instance information, proving load balancing is working.

üîç Validation Checklist

  1. Launch Template: Created with correct AMI, instance type, and user data
  2. Auto Scaling Group: Running with 2 instances across multiple AZs
  3. Load Balancer: Active and accessible via DNS name
  4. Target Group: Shows healthy instances from ASG
  5. Web Application: Displays different instance info on refresh
  6. Scaling Policy: Configured to scale based on CPU utilization
